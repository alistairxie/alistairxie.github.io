（一）Activity 基础概念

Activity 是 Android 应用的可视化界面载体，每个屏幕显示的内容基本对应一个 Activity（如登录界面→LoginActivity、首页→MainActivity），多个 Activity 通过 Intent 实现跳转和数据传递，Activity 的管理依赖 “任务栈（Task Stack）”—— 遵循 “先进后出” 原则，新启动的 Activity 压入栈顶，按下返回键时栈顶 Activity 出栈，直至栈空应用退出。

（二）Activity 生命周期

  • 1.核心回调方法及作用
方法名 调用时机 核心作用
onCreate() Activity 首次创建时调用（仅一次） 初始化布局（setContentView (R.layout.activity_main)）、绑定控件、初始化数据（如读取 SP 存储、初始化网络请求工具）
onStart() Activity 由 “不可见” 变为 “可见” 时调用 执行与界面可见相关的操作（如启动动画、注册广播接收器）
onResume() Activity 获取焦点（可与用户交互）时调用 启动定时器、开启传感器监听（如定位、加速度传感器）、恢复媒体播放
onPause() Activity 失去焦点（仍可见，如被弹窗覆盖）时调用 暂停定时器、停止传感器监听、保存临时数据（避免数据丢失）
onStop() Activity 完全不可见时调用 释放非必要资源（如关闭数据库连接、停止网络请求）、取消广播注册
onDestroy() Activity 销毁时调用（仅一次） 释放所有资源（如关闭文件流、取消服务绑定）、避免内存泄漏
onRestart() Activity 由 “销毁状态” 重新启动前调用 恢复 Activity 重启前的状态（如恢复列表滚动位置）

  • 2.生命周期流程示例：打开 MainActivity→跳转至 SecondActivity→返回 MainActivity→关闭应用

  • MainActivity：onCreate () → onStart () → onResume ()（显示并可交互）

  • 跳转 SecondActivity：MainActivity.onPause () → SecondActivity.onCreate () → SecondActivity.onStart () → SecondActivity.onResume () → MainActivity.onStop ()

  • 返回 MainActivity：SecondActivity.onPause () → MainActivity.onRestart () → MainActivity.onStart () → MainActivity.onResume () → SecondActivity.onStop () → SecondActivity.onDestroy ()

  • 关闭应用：MainActivity.onPause () → MainActivity.onStop () → MainActivity.onDestroy ()

  • 3.遇到的问题与解决
    AlertDialog为什么不会触发onPause()？

  • Activity的前台状态保持不变：当AlertDialog显示时，它通常是通过覆盖当前Activity的部分或全部UI来实现的，这意味着用户仍然与同一个Activity交互。

  • 生命周期回调：在Android中，onPause()方法是在Activity即将失去焦点并被另一个Activity覆盖时调用的。由于AlertDialog并没有导致Activity失去焦点，所以不会触发onPause()。

    • startActivity(intent)与getIntent()无法实现 Activity 双向数据传递

          graph TD;
A[Activity A]-->|startActivity|B[Activity B];
B[Activity B]-->|startActivity|A[Activity A];
      • startActivity(intent)用于启动新 Activity 并传递数据（单向从启动方到被启动方），而getIntent()仅能获取启动当前 Activity 的原始 Intent，二者组合无法实现两个 Activity（如 A 和 B）之间的双向数据传递。

    • 根本原因

      • getIntent()的设计逻辑是 “获取启动当前 Activity 的初始 Intent”，其数据仅来源于首次启动该 Activity 时传入的 Intent。若后续其他 Activity（如 B）试图通过intent.putExtra()向其传递新数据，原 Activity（如 A）的getIntent()无法感知或更新这些新数据 —— 因为getIntent()不支持动态接收后续传入的 Intent 数据，仅绑定首次启动时的原始 Intent。

    • 分启动模式实例验证

      • Activity 的启动模式（launchMode）会影响实例创建逻辑，进一步暴露双向传递的失效场景：
        • 1.标准模式（standard，默认）：看似可行，实则数据混乱
          • 现象：A 启动 B 时，A 通过intent.putExtra("data", "A的数据")向 B 传递数据，B 可通过getIntent()获取；B 试图通过intent.putExtra("reply", "B的回复")并startActivity(intent)启动 A 时，A 表面上能通过getIntent()获取 “B 的回复”。
          • 本质：standard 模式下，每次startActivity()都会创建新的 Activity 实例。此时 A 获取的 “B 的回复”，实际是新创建的 A 实例从启动它的 B 那里获取的数据，而非原来的 A 实例。两个 A 实例是相互独立的，原始 A 实例始终无法获取 B 的回复。
        • 2.单实例模式（singleInstance）：直接失效
          • 现象：若 A 的启动模式为singleInstance（整个系统中仅存在一个 A 实例），B 通过intent.putExtra("reply", "B的回复")并startActivity(intent)启动 A 时，A 调用getIntent()获取的仍是首次启动时的原始数据，永远无法得到 B 传递的新数据。
          • 本质：singleInstance 模式下，系统会复用已存在的 A 实例，而getIntent()绑定的是 A 首次启动时的 Intent，后续 B 传入的新 Intent 不会更新到已有 A 实例的getIntent()中，导致双向传递完全失效。

（四）任务栈（Task Stack）概念与应用

  1. 核心特性：每个应用默认对应一个任务栈，栈中存放该应用启动的所有 Activity；当用户点击桌面图标启动应用时，若任务栈不存在则创建新栈，将根 Activity（启动页或首页）压入栈；后续跳转的 Activity 依次压入栈顶，返回时栈顶 Activity 出栈，直至栈空，应用进程进入后台（若系统内存不足，进程可能被销毁）。
  2. 常见启动模式（通过 AndroidManifest.xml 中 Activity 的 launchMode 属性设置，解决重复创建 Activity、控制栈中 Activity 实例等问题）
    • standard（默认模式）：每次启动 Activity 都会创建新实例，无论栈中是否已存在，压入当前任务栈顶（如多次跳转同一 Activity，栈中会有多个实例）。
    • singleTop（栈顶复用）：若目标 Activity 已在栈顶，则直接复用，调用 onNewIntent () 方法接收新 Intent；若不在栈顶，则创建新实例压入栈顶（适用于通知跳转、首页跳转首页等场景，避免重复创建）。
    • singleTask（栈内唯一）：若任务栈中已存在目标 Activity 实例，则将该实例之上的所有 Activity 出栈，使目标 Activity 成为栈顶（调用 onNewIntent ()）；若不存在，则创建新实例压入栈（适用于首页、个人中心等需唯一实例的场景）。
    • singleInstance（单独任务栈）：目标 Activity 独占一个新任务栈，且该任务栈中只有这一个 Activity 实例；其他应用启动该 Activity 时，直接复用已存在的实例（适用于系统级应用或需全局唯一的页面，如电话、地图定位页面）。
