AndroidManifest.xml是 Android 应用的全局配置清单文件，位于项目app/src/main目录下，相当于应用的 “身份证” 与 “说明书”。系统通过它识别应用的包名、组件（Activity/Service 等）、权限、版本信息等关键配置，无此文件或配置错误会导致应用无法安装或运行。本文将从核心作用、结构解析、关键节点配置及实战技巧等维度，全面讲解其使用方法。

一、核心作用：为什么 AndroidManifest.xml 必不可少？

AndroidManifest.xml的核心价值是 “向系统声明应用的关键信息”，具体包含以下 5 大核心作用：

  1. 标识应用唯一性：通过package属性定义应用唯一包名（如com.example.myapp），是应用在系统中的唯一标识（与build.gradle的applicationId共同决定应用安装、更新的唯一性）；

  2. 声明应用组件：注册应用的所有组件（Activity、Service、BroadcastReceiver、ContentProvider），系统需通过清单感知组件存在才能启动；

  3. 申请系统权限：声明应用需要的系统权限（如网络、相机、存储权限），部分高危权限还需在代码中动态申请，但清单声明是前提；

  4. 配置应用基础信息：定义应用图标、名称、主题、最小支持 SDK 版本、目标 SDK 版本等，直接影响应用的外观与兼容性；

  5. 定义组件间跳转规则：通过intent-filter配置组件的跳转协议（如主 Activity 的启动规则、自定义 Scheme 跳转），实现组件间或应用间的通信。

二、基础结构：AndroidManifest.xml 的骨架

AndroidManifest.xml遵循 XML 语法规范，根节点为<manifest>，核心子节点包含<application>（应用全局配置）、<uses-permission>（权限申请）、<uses-sdk>（SDK 版本配置，已逐步被build.gradle替代）等。以下是一个完整的基础模板：

三、关键节点深度解析：每个配置的 “潜台词”

3.1 根节点<manifest>：全局配置入口

核心属性解析：

属性名 作用 注意事项
package 定义应用包名（唯一标识） 与build.gradle的applicationId需一致（Android Studio Arctic Fox + 后，namespace替代其作为 R 类包名）
android:versionCode 版本号（整数） 用于应用商店判断更新（如 1→2 表示升级），必须递增，用户不可见
android:versionName 版本名称（字符串） 显示给用户（如 “1.0.0”“1.1.0-beta”），可自定义格式
xmlns:android 引入 Android 命名空间 固定值http://schemas.android.com/apk/res/android，否则无法使用android:前缀的属性

3.3 应用配置节点<application>：全局行为控制

核心属性解析（影响整个应用的运行与外观）：

属性名 作用 常用值与注意事项
android:name 自定义 Application 类的完整路径 需继承android.app.Application，用于初始化全局资源（如第三方库），不配置则使用系统默认 Application
android:icon 应用图标 引用mipmap目录下的图片（如@mipmap/ic_launcher），Android Studio 推荐优先使用mipmap而非drawable
android:label 应用名称（桌面图标名称 + 默认 Activity 标题） 引用values/strings.xml的资源（如@string/app_name），避免硬编码
android:theme 应用全局主题 控制 Activity 的外观（如是否显示 ActionBar、窗口背景色），常用主题如Theme.MaterialComponents.DayNight.NoActionBar
android:allowBackup 是否允许系统备份应用数据 默认true，若需禁止备份（如含敏感数据）设为false
android:exported 是否允许外部应用访问应用组件 Android 12 + 新增的全局属性，可统一控制所有组件的exported（组件单独设置会覆盖全局）
android:requestLegacyExternalStorage Android 10 + 是否使用旧存储模式 Android 10 默认启用分区存储，设为true可兼容旧版存储逻辑（Android 11 + 需用MANAGE_EXTERNAL_STORAGE权限）

3.4 组件注册节点：Activity/Service/Receiver/Provider

3.4.1 <activity>：Activity 注册（最常用）

每个 Activity 必须在此注册，否则系统无法启动（动态创建的 Activity 也需注册），核心属性：

属性名 作用 关键注意事项
android:name Activity 完整类名 可省略包名前缀（如.ui.MainActivity等效于com.example.myapp.ui.MainActivity）
android:exported 是否允许外部应用启动此 Activity Android 12 + 必填！主 Activity（含LAUNCHER过滤器）需设为true，内部 Activity 设为false（安全）
android:launchMode Activity 启动模式 控制 Activity 实例的创建规则（如standard、singleTop、singleTask、singleInstance），解决页面重复创建问题
android:screenOrientation 屏幕方向 portrait（竖屏，固定）、landscape（横屏，固定）、unspecified（跟随系统，默认）
android:finishOnTaskLaunch 重新打开应用时是否关闭此 Activity true表示任务栈重建时关闭该 Activity（如退出应用后重新打开，不保留之前的页面）

3.4.2 <service>：Service 注册

Service 是后台运行的组件，需注册才能被启动，核心属性：

  • android:exported：是否允许外部应用启动此 Service（内部服务设为false，安全）；

  • android:foregroundServiceType：Android 12 + 前台服务必须指定类型（如dataSync（数据同步）、location（定位）），否则崩溃；

  • android:stopWithTask：当应用任务栈被清除时，是否停止此 Service（true表示跟随任务停止，默认false）。

3.4.4 <provider>：ContentProvider 注册

ContentProvider 用于跨应用数据共享（如读取系统联系人、自定义数据共享），核心属性：

  • android:authorities：唯一标识（跨应用访问时需指定此值，格式如com.example.myapp.provider）；

  • android:readPermission/android:writePermission：控制读写权限（仅拥有权限的应用可访问）；

  • android:exported：是否允许外部应用访问（若仅内部使用设为false）。

四、实战避坑指南：这些错误 90% 的开发者都踩过

4.1 错误 1：Activity 未注册或exported配置错误（Android 12+）

  • 现象：应用启动崩溃，日志提示 “Activity not found” 或 “Targeting S+ (version 31 and above) requires that an explicit value for android:exported be defined when intent filters are present”；

  • 原因：

  1. Activity 未在<application>中注册；

  2. Android 12+（SDK 31+）中，含<intent-filter>的 Activity 未设置android:exported属性；

  • 解决：

  1. 确保所有 Activity 都有<activity>节点注册；

  2. 含<intent-filter>的 Activity（如主 Activity）设android:exported="true"，内部 Activity（无<intent-filter>）设android:exported="false"。

4.2 错误 2：权限申请遗漏或版本不匹配

  • 现象：调用需要权限的 API 时崩溃（如网络请求提示 “Permission denied”，相机调用失败）；

  • 原因：

  1. 未在清单中声明对应权限；

  2. Android 6.0+（SDK 23+）高危权限未动态申请；

  3. Android 13 + 使用了旧权限名称（如存储权限用READ_EXTERNAL_STORAGE而非READ_MEDIA_IMAGES）；

  • 解决：

  1. 清单中声明正确的权限（参考Android 官方权限列表）；

  2. 高危权限通过ActivityResultContracts动态申请（替代旧的requestPermissions方法）；

  3. 根据targetSdkVersion调整权限名称（如targetSdk=33时用READ_MEDIA_IMAGES）。

4.3 错误 3：包名与applicationId不一致

  • 现象：应用安装失败，提示 “App not installed”，或应用商店上传时提示 “包名已存在”；

  • 原因：AndroidManifest.xml的package与build.gradle的defaultConfig.applicationId不一致（Android Studio Arctic Fox + 后，namespace需与applicationId一致）；

4.4 错误 4：Application类配置错误

  • 现象：应用启动崩溃，日志提示 “ClassNotFoundException: Didn’t find class “com.example.myapp.MyApplication””；

  • 原因：

  1. android:name属性配置的类名错误（如包名拼写错误、类未创建）；

  2. 自定义Application类未继承android.app.Application；

  • 解决：

  1. 核对android:name的类名（如.MyApplication对应com.example.myapp.MyApplication）；
