一、Git 是什么？—— 理解版本控制的核心价值

  1. 本质定位：Git 是一款开源的分布式版本控制系统，用于跟踪文件（尤其是代码）的修改历史，支持多人协作开发，核心解决 “代码版本混乱、多人修改冲突、历史版本回溯” 三大问题。

  2. 与集中式版本控制（如 SVN）的区别：

    • 集中式（SVN）：所有版本数据存储在中央服务器，本地仅保留当前工作文件，断网无法查看历史或提交；

    • 分布式（Git）：每个开发者本地都有完整的版本库，支持离线提交，仅在需要同步时与远程仓库交互（如 GitHub、GitLab），协作更灵活。

  3. 核心优势：

    • 轻量级：操作速度快，占用资源少；

    • 安全可靠：本地完整备份，避免中央服务器故障导致数据丢失；

    • 分支灵活：支持快速创建、合并分支，隔离不同开发任务（如功能开发、bug 修复）。

二、Git 环境配置 —— 从安装到身份初始化

1. 安装 Git（跨平台支持）

  • Windows：下载Git 官方安装包，按向导安装（建议勾选 “Add Git to PATH”，方便全局调用），安装完成后打开 “Git Bash” 验证；

  • macOS：通过 Homebrew 安装（brew install git），或直接下载官方包；

  • Linux：通过包管理器安装（Ubuntu：sudo apt-get install git；CentOS：sudo yum install git）。

  • 验证安装：终端输入git --version，显示版本号（如git version 2.45.1）即成功。

三、Git 核心概念 —— 理解 “工作区、暂存区、本地仓库” 模型

Git 的核心是 “三区域” 数据流转，所有操作围绕这三个区域展开，必须先理解其逻辑关系：

  1. 工作区（Working Directory）：即本地电脑中可见的项目文件夹，日常编写代码的地方，文件状态分为 “未跟踪（Untracked）”“已跟踪（Tracked）”；

  2. 暂存区（Staging Area/Index）：临时存放待提交的修改，是工作区与本地仓库的 “中间层”，通过git add将工作区修改添加到暂存区；

  3. 本地仓库（Local Repository）：存储完整的版本历史，通过git commit将暂存区的修改提交到本地仓库，形成一个不可修改的 “版本快照”；

  4. 远程仓库（Remote Repository）：用于多人协作的共享仓库（如 GitHub 上的仓库），通过git push/pull与本地仓库同步数据。

数据流转示意图：

工作区修改 → git add → 暂存区 → git commit → 本地仓库 → git push → 远程仓库

远程仓库 → git pull → 本地仓库 → git checkout → 工作区

五、Git 分支操作 —— 隔离开发任务的核心

分支是 Git 的 “灵魂功能”，通过分支可并行开发不同任务（如 “开发新功能” 和 “修复线上 bug” 互不干扰），核心原则：“主分支（main）保持稳定，功能开发在分支上进行，完成后合并回主分支”。

1. 分支核心概念

  • 主分支（main/master）：默认创建的分支，用于存放正式版本的代码，禁止直接在主分支开发；

  • 功能分支（feature/xxx）：用于开发新功能（如feature/playlist），从 main 分支创建，完成后合并回 main；

  • 修复分支（hotfix/xxx）：用于修复线上 bug（如hotfix/play-crash），从 main 分支创建，修复后合并回 main 和开发分支。

八、Git 常见问题与解决

  1. “git push 报错：fatal: Could not read from remote repository”

    原因：远程仓库地址错误，或本地无权限访问（如 GitHub 未配置 SSH 密钥）。

    解决：

    • 检查远程地址：git remote -v，确认地址正确；

    • 若用 HTTPS 协议，推送时需输入 GitHub 账号密码（或使用个人访问令牌）；

    • 推荐配置 SSH 密钥（免密码登录）：生成 SSH 密钥（ssh-keygen -t rsa -C "``your.email@example.com``"），将~/.ssh/id_rsa.pub的内容添加到 GitHub 的 SSH 密钥列表。

  2. “git merge 后想撤销合并（合并错误）”

    解决：若合并后未提交，直接执行git merge --abort撤销合并；若已提交，找到合并前的版本号，执行git reset --hard 版本号（注意：硬回溯会丢失合并后的修改，需谨慎）。

  3. “误删本地分支，如何恢复？”

    解决：通过git reflog查看分支的删除记录，找到删除前的版本号，执行git checkout -b 恢复的分支名 版本号，即可重建分支。

九、Git 学习总结 —— 核心流程与注意事项

  1. 日常开发核心流程：

    拉取远程代码（git pull）→ 本地开发 → 暂存修改（git add）→ 提交版本（git commit）→ 推送远程（git push）

  2. 分支协作原则：

    • 主分支（main）保持稳定，仅合并经过测试的代码；

    • 功能开发在 feature 分支，bug 修复在 hotfix 分支，避免直接修改主分支；

    • 合并分支前先拉取远程代码，解决冲突后再提交。

  3. 关键注意事项：

    • 提交信息需清晰规范，便于后续回溯和协作；

    • 敏感信息（如密钥、密码）禁止提交到 Git 仓库，需通过.gitignore排除；

    • 定期备份远程仓库，避免本地仓库损坏导致数据丢失。

    通过以上内容，可掌握 Git 的日常开发必备操作，后续结合 Repo 工具（管理多仓库），即可应对从个人小型项目到大型团队协作的版本控制需求。
